Purpose

This document explains when to use generated JWT tokens and how to test available endpoints quickly.

Tokens Generated By Auth APIs

After successful register or login, the API returns:

  • accessToken
  • refreshToken

When To Use These Tokens

Access Token

Use this token when calling protected APIs.

How to send it:

  • Header: Authorization: Bearer <accessToken>

Use cases:

  • Employee endpoints
  • Leave endpoints
  • Attendance endpoints
  • Reporting endpoints

Refresh Token

Use this token only when the access token expires.

Use cases:

  • Request a new access token without forcing user login again
  • Session continuity

Typical Flow

  1. User calls register or login
  2. Backend returns accessToken and refreshToken
  3. Client calls protected APIs with accessToken
  4. If access token expires, client uses refreshToken to obtain a new one
  5. Continue API calls with the new access token

Endpoint Testing With Curl Script

A curl script already exists at docs/curls.sh.

Open Curl Commands Page

What it currently covers:

  • POST /api/v1/auth/register
  • POST /api/v1/auth/login
  • GET /actuator/health
  • GET /actuator/info
  • H2 console access details

How to run:

chmod +x docs/curls.sh
    ./docs/curls.sh

Local Base URL

Use local loopback URL for reliable testing:

  • http://127.0.0.1:8080

Run With Redis Security State (Local)

Use the helper script for one-command startup:

./docs/run-local-redis.sh

This command starts Redis and runs the app with local-redis profile where:

  • security.redis.enabled=true
  • token revocation and lockout state are stored in Redis

Stop Redis service when done:

./docs/stop-local-redis.sh
