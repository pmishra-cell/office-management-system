This page explains what happens at runtime from request entry to security checks and state storage.

Application Startup

  1. Spring Boot starts and loads application.yml.
  2. Security chain is built in SecurityConfig.
  3. Store beans are selected by security.redis.enabled.
  4. JPA initializes the users table from UserAccount.

Request Pipeline

  1. HTTP request reaches Spring MVC.
  2. JwtAuthenticationFilter runs before username/password filter.
  3. If bearer token exists and is valid access token, auth is loaded into SecurityContext.
  4. Controller method executes.
  5. On errors, ApiExceptionHandler converts exceptions to JSON responses.

Auth Flows

Register

  1. POST /api/v1/auth/register
  2. Duplicate email check via repository.
  3. Password hashed with BCrypt.
  4. User saved and token pair returned.

Login

  1. LoginAttemptService.assertLoginAllowed enforces lockout policy.
  2. Credential check against stored password hash.
  3. Success clears attempt state; failure increments and can lock account.

Refresh and Logout

  1. Refresh token must have claim type=refresh.
  2. Revocation store is checked.
  3. Refresh rotates token and revokes previous one.
  4. Logout revokes the supplied refresh token.

Role Authorization

  • /api/v1/secure/me: authenticated users
  • /api/v1/secure/employee: EMPLOYEE, MANAGER, ADMIN
  • /api/v1/secure/manager: MANAGER, ADMIN
  • /api/v1/secure/admin: ADMIN only

Security State Storage

In-Memory Mode

Enabled when security.redis.enabled=false (default). State lives in process memory and is lost on restart.

Redis Mode

Enabled when security.redis.enabled=true. State is shared through Redis keys:

  • {key-prefix}:login-attempts:{normalized-email}
  • {key-prefix}:revoked-refresh:{refresh-token}
